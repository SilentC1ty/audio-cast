# 🚀 AudioCast 产品需求文档 (PRD)

## 1. 产品概述与需求定义

### 1.1 产品愿景
开发一款局域网音频转发工具，将 Android 设备的全局音频（系统声音、媒体、游戏）无线串流至桌面端播放。满足用户在电脑前工作或娱乐时，统一使用电脑音响/耳机收听双端音频的需求，消除频繁切换耳机的繁琐。

### 1.2 核心需求 (User Stories)
*   **无缝发现与连接：** 用户打开手机 App 和电脑客户端后，App 能自动发现局域网内的电脑，一键建立连接，无需手动输入 IP。
*   **全局音频截取：** 支持捕获 Android 10+ 系统的全局媒体音频（排除不支持录屏的受保护内容，如 DRM 视频）。
*   **视听同步 (Lip-sync)：** 传输延迟需控制在 **80ms - 120ms** 之间，确保用户在手机上看视频时，电脑端播放的声音与手机画面不产生明显的违和感。
*   **后台稳定运行：** 手机息屏或 App 切入后台时，音频转发不能中断。

---

## 2. 开发重点与技术难点 (优先级排序)

为了达成上述需求，特别是 100ms 左右的视听同步要求，开发过程中必须严格把控以下核心节点：

### 🔴 重点 1：绕过 UI 框架的音频管线 (性能生命线)
*   **雷区：** 绝对不能将原生捕获的 PCM 音频数据通过 Flutter 的 `MethodChannel` 传给 Dart 线程，再由 Dart 发送网络请求。高频的数据序列化会导致严重的 CPU 占用和延迟剧增。
*   **正确路径：** Flutter 只做 UI 控制器。在 Android 原生层（Kotlin 或 C++ JNI）建立完整的闭环：`AudioRecord 捕获 -> Opus 编码器压缩 -> UDP Socket 发送`。

### 🔴 重点 2：接收端的抗抖动缓冲 (Jitter Buffer)
*   **挑战：** 局域网 Wi-Fi 存在固有的信号抖动（Jitter），UDP 包到达时间不均匀。如果没有缓冲区，声音会频繁“破音”或“电音”。
*   **方案：** 在 Tauri 的 Rust 后端实现一个长度约为 `60ms - 80ms` 的动态缓冲区。先让音频包在队列里“飞一会”再交给操作系统的音频接口播放。这能用可接受的轻微延迟换取极高的声音平滑度。

### 🟡 重点 3：Android 后台保活机制
*   **挑战：** 录音是一个持续耗电的行为，极易被 Android 系统的电源管理（Doze Mode）杀掉。
*   **方案：** 必须在开启音频流转的同时，在 Kotlin 层启动一个带有明确通知栏提示的 **前台服务 (Foreground Service)**，声明 `foregroundServiceType="mediaProjection"`。

## 3. 技术架构选型

### 3.1 移动端 (发送端)
*   **UI 与应用架构：** **Flutter (Dart)** - 负责构建美观的跨平台 UI、状态管理以及与原生层的信令交互。
*   **系统交互层：** Android Native (Kotlin)
    *   `MediaProjection API`：获取截取系统屏幕/音频的系统级授权。
    *   `AudioPlaybackCapture API`：Android 10 引入的专用内录 API。
*   **音频处理层：** C++ (通过 `dart:ffi` 或 JNI 调用) 运行 Opus 编码库。

### 3.2 桌面端 (接收端)
*   **UI 渲染层：** **Vue 3 + Vite + TailwindCSS** (运行在 Tauri 的 WebView 中)。
*   **系统宿主层：** **Tauri (Rust)** - 提供极小的包体积和极低的内存占用。
*   **音频引擎层：**
    *   网络：Rust 标准库 `std::net::UdpSocket`。
    *   解码：Rust `opus` crate。
    *   输出：Rust `cpal` crate (直接调用 WASAPI/CoreAudio 实现低延迟播放)。

## 4. 通信与数据协议

### 4.1 控制信令 (TCP / mDNS)
*   **发现机制：** 桌面端通过 mDNS (Zeroconf) 广播服务，Flutter 端使用 `multicast_dns` 或原生插件扫描。
*   **握手协议：** 建立连接前，通过 TCP 短连接发送配置信息（如：`{"action": "start", "sampleRate": 48000, "channels": 2}`），桌面端返回 UDP 接收端口。

### 4.2 媒体流传输 (UDP)
*   **格式：** 48kHz, 16-bit PCM, 经过 Opus VBR 编码。
*   **封包策略：** 帧长设为 10ms 或 20ms。包头需包含 `Sequence Number`（序列号，用于接收端丢弃乱序包）和 `Timestamp`（时间戳，用于缓冲区对齐）。

## 5. UI/UX 设计规范

### 5.1 移动端 (Flutter)
*   **状态导向：** 首页中心显示醒目的状态指示器（未连接 / 搜索中 / 流转中）。
*   **操作极简：** 底部提供抽屉式的“可用设备列表”，点击即连。
*   **网络提示：** 在连接状态下，左上角或右上角显示实时的网络延迟（通过心跳包计算）和丢包率，以便用户排查 Wi-Fi 质量问题。

### 5.2 桌面端 (Tauri + Vue)
*   **小巧克制：** 默认以类 Widget 的小窗口形式停靠在屏幕边缘。
*   **控制中心：**
    *   提供独立的音量滑块（不影响电脑主系统音量）。
    *   提供一键静音按钮。
    *   **高级设置：** 暴露“缓冲区大小 (Buffer Size)”调节滑块，允许高级用户在“极低延迟”和“抗干扰稳定”之间手动微调。
